# H14.13 Baseline v1 Playbook

## Purpose
`baseline-v1.json` is the frozen curated fixture baseline used by WoS parity reconciliation (`H14.13`).
It is manual/versioned and must not be regenerated automatically.

## Source of truth
- Canonical source: one approved **staging** H14.12 full-run snapshot.
- Source run metadata is tracked in:
  - `src/main/resources/wos/parity/baseline-v1.metadata.json`

## How baseline v1 is curated
1. Select exactly 20 journals that cover:
   - AIS/RIS present,
   - IF present + IF missing,
   - SCIE + SSCI,
   - bundled split cases,
   - `SCIENCE -> SCIE` normalization cases.
2. Freeze those `journalId`s and tags in metadata.
3. Fill `baseline-v1.json` with expected values from approved staging run:
   - counts,
   - edition counters,
   - timeline/category/scoring/if-missing checks.
4. Keep `allowlistedMismatches` empty by default.

## Re-baselining policy
Re-baselining is allowed only when one of these changes occurs:
- parser/normalization contract change,
- source policy change,
- approved migration schema/behavior change.

Re-baseline steps:
1. Copy `baseline-v1.json` to new version (`baseline-v2.json`).
2. Update metadata with new source run id/date and rationale.
3. Keep old baseline file for audit/history.
4. Review diff with focus on:
   - count deltas,
   - edition normalization deltas,
   - scoring behavior deltas.

## Validation
Run:
- `npm run verify-wos-parity-baseline`

This checks JSON integrity, required keys, fixture size (20), and coverage tag presence.

## Current v1 caveat
The currently generated `v1` baseline reflects available canonical facts in the active dataset.
If IF/RIS facts are not present yet, metadata must explicitly record this as a coverage gap and `v2` re-baselining is required after those facts are available.
