# H22.7 Operationalization, Rollback, and Rebuild Playbook

Date: 2026-03-12  
Status: implemented.

## Summary
H22.7 operationalizes first-wave H22 Postgres reads with:
- a strict gate-driven rollback policy,
- deterministic rebuild and recovery steps,
- a consolidated operational status surface.

Policy lock:
- rollback is mandatory when parity/performance gate fails, or when projection/materialized refresh status is `FAILED` during cutover operations.

## Pre-Cutover Checklist
Execute in order from `/admin/initialization`:
1. Run Postgres projection full rebuild (`/postgres/projection/runFull`).
2. Run projection incremental sync (`/postgres/projection/runIncremental`) and confirm no hard failures.
3. Refresh all materialized views (`/postgres/materialized/refreshAll`).
4. Run dual-read gate (`/postgres/dualReadGate/run`).
5. Confirm consolidated operational status (`/postgres/operational/showStatus` or `/postgres/operational/status`).

Required readiness before cutover:
- projection latest run: `SUCCESS`,
- materialized-view refresh latest run: `SUCCESS`,
- dual-read gate latest run: `SUCCESS`,
- read-store currently `mongo` (pre-cutover baseline).

## Cutover Execution
1. Set `app.reporting.read-store=postgres` (for example `APP_REPORTING_READ_STORE=postgres`).
2. Restart application.
3. Validate startup guard outcome:
   - app starts cleanly,
   - `PostgresReadCutoverGuard` passes,
   - consolidated operational status is `GREEN`.
4. Run post-start checks:
   - `/admin/initialization/postgres/operational/status`,
   - `/admin/initialization/postgres/projection/status`,
   - `/admin/initialization/postgres/materialized/status`,
   - `/admin/initialization/postgres/dualReadGate/status`.

## Strict Rollback Triggers
Rollback immediately when any of the following is true during cutover/rehearsal window:
- dual-read gate latest run is `FAILED` (parity mismatch or p95 threshold breach),
- projection latest run is `FAILED`,
- materialized-view refresh latest run is `FAILED`,
- consolidated operational status is `RED` while read-store is `postgres`.

## Rollback Execution
1. Set `app.reporting.read-store=mongo` (for example `APP_REPORTING_READ_STORE=mongo`).
2. Restart application.
3. Validate rollback:
   - reporting/scopus/admin read paths resolve via Mongo,
   - consolidated operational status no longer indicates postgres-required red condition,
   - initialization status endpoints remain reachable.
4. Log rollback incident with failing run ids (`projection`, `mv refresh`, `dual-read gate`).

No data rollback is required (read-store routing rollback only).

## Deterministic Rebuild And Recovery
Use this sequence after rollback or failed rehearsal:
1. Optional reset (only when a destructive reset is explicitly intended):
   - projection checkpoints reset: `/postgres/projection/resetState` with `RESET`.
2. Run projection full rebuild.
3. Run projection incremental sync.
4. Refresh all materialized views.
5. Run dual-read gate.
6. Check consolidated operational status and require `YELLOW` (mongo mode, no failures) before next cutover attempt.

If any step fails:
- stop sequence,
- triage failed component first,
- rerun from the failed step once corrected.

## Incident Triage Map
Use this priority order:
1. Consolidated status (`/postgres/operational/status`) for overall state and failing component.
2. Projection status (`/postgres/projection/status`) for slice/checkpoint drift or failed run details.
3. MV refresh status (`/postgres/materialized/status`) for heavy-read refresh failures.
4. Dual-read gate status (`/postgres/dualReadGate/status`) for parity/performance mismatch samples.
