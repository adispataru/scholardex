# H22.1 Postgres Reporting Architecture Contract

Date: 2026-03-11  
Status: active contract baseline for H22 migration.

## Summary

This document locks the H22.1 architecture contract for migrating WoS/scoring/reporting/admin read surfaces to PostgreSQL while keeping MongoDB as the ingestion/event write origin.

This is a contract/documentation slice only. H22.2+ implement schema, projector, cutover, and runtime rewires based on this contract.

## Decision Locks

- Read cutover strategy is `big-bang` for first-wave migrated read surfaces.
- Migration strategy is `reset-first` in current environment:
  - WoS/Scopus-derived read datasets may be dropped and rebuilt.
  - no long-lived dual-authoritative reconciliation is required.
- Post-cutover runtime read authority for migrated surfaces is PostgreSQL.
- MongoDB remains authoritative for ingestion/event/canonical write origin.
- First-wave scope includes WoS/scoring/reporting and admin read surfaces.

## Bounded Contexts

### A. Mongo Write/Ingest Context (authoritative write origin)

Purpose:
- own ingest events, touch queues, source lineage, and canonical ingest-side facts as currently modeled.

Owned concerns:
- source payload/event ingestion (`SCOPUS`, `WOS`, other supported sources),
- queue/touch processing and ingest orchestration metadata,
- canonical ingest lineage used to build downstream read models.

Contract:
- this context is write-origin and projection source for H22.
- it is not the post-cutover runtime read authority for migrated H22 read surfaces.

### B. Postgres Reporting-Read Context (authoritative runtime read context post-cutover)

Purpose:
- serve migrated WoS/scoring/report/admin read paths with SQL joins/materialized views.

Owned concerns:
- WoS ranking/scoring read models,
- citation-heavy reporting read models,
- admin/reporting query shapes currently assembled through multi-collection Mongo reads.

Contract:
- read-optimized, derived from Mongo write/ingest context through projector/backfill flows.
- post-cutover, this context is authoritative for migrated runtime reads.

## Ownership Rules

### During transition (H22.2-H22.4 pre-cutover)

- Mongo remains ingest/write origin.
- PostgreSQL is populated via rebuild/backfill/projector pipelines.
- no permanent dual-authoritative read-write mode is allowed.
- temporary read verification (dual-read parity) is allowed only for validation gates.

### Post-cutover (H22.4+ for migrated surfaces)

- PostgreSQL serves all migrated runtime reads.
- Mongo-backed read paths for migrated surfaces are transitional debt and must be removed/disabled by cutover completion.
- Mongo continues as source ingest/event/canonical write origin for projector input.

## Data-Flow Contract

Canonical flow:
1. Source ingest writes canonical/event data to Mongo context.
2. Projector/backfill pipeline derives PostgreSQL read models.
3. Runtime migrated read surfaces query PostgreSQL only (post-cutover).

Backfill/rebuild contract:
- rebuild must be deterministic for the same source Mongo snapshot.
- rerun must converge to equivalent PostgreSQL read state (idempotent from a state perspective).
- reset-first execution is allowed for local migration rehearsal and first-wave cutover preparation.

Freshness contract:
- H22.1 does not lock sync cadence policy (near-real-time vs batch); cadence is defined in H22.3.
- H22.1 locks direction only: `Mongo -> Postgres` for migrated read models.

## Compatibility Contract

### First-wave read surfaces that must switch together at big-bang cutover

- Reporting lookup/scoring read contracts behind `ReportingLookupPort` and its runtime implementation(s).
- WoS ranking/scoring read assembly paths used in reporting/scoring services.
- Admin/reporting read facades and query services that currently assemble WoS/scopus/reporting reads:
  - `AdminInstitutionReportFacade`
  - `ScholardexAdminReadFacade`
  - `ScholardexAuthorQueryService`
  - `ScholardexForumQueryService`
  - `ScholardexAffiliationQueryService`
  - `ProjectionBackedReportingLookupFacade`

### Temporary compatibility allowed before switch

- dual-read verification for parity/performance gates is allowed.
- production runtime fallback to Mongo for migrated surfaces is not a permanent operating mode and must not remain enabled after cutover acceptance.

## API and Interface Freeze (for H22.4 target ownership)

- H22.1 introduces no runtime code API changes.
- Target ownership is frozen:
  - migrated WoS/scoring/report/admin runtime read interfaces must resolve from PostgreSQL post-cutover.
  - Mongo-backed implementations for those read interfaces are transitional only.
- Source-of-truth freeze:
  - Mongo = ingest/event/canonical write origin.
  - Postgres = authoritative runtime read store for migrated surfaces post-cutover.

## Out of Scope for H22.1

- SQL schema/table/index DDL finalization (`H22.2`).
- Projector/backfill implementation details (`H22.3`).
- Runtime code cutover and query rewiring (`H22.4+`).
- Materialized-view physical design and refresh scheduling (`H22.5`).
- Parity/performance gate thresholds and automation wiring (`H22.6`).
- Operational runbook execution details (`H22.7`).
