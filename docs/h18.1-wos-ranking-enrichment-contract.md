# H18.1 WoS Ranking Enrichment Computation Contract

## Summary
This contract locks deterministic computation rules for missing `rank`, `quarter` (quartile label), and `quartileRank` in canonical WoS category facts.

This is a policy/documentation slice only. Runtime implementation is in H18.2+.

## Canonical Scope and Join Key
- Canonical storage remains:
  - `wos.category_facts` (`quarter`, `rank`, `quartileRank`)
  - `wos.metric_facts` (`value`)
- Enrichment grouping key:
  - `year + metricType + categoryNameCanonical + editionNormalized`
- Candidate records for one group:
  - category facts in that group, joined to same-journal metric fact on:
    - `journalId + year + metricType`

## Field Precedence and Preservation
- Field-level precedence is mandatory:
  - if source value exists, preserve it;
  - compute only missing fields.
- `quarter` precedence:
  - source `quarter` always wins and is never overwritten by computed quartile buckets.
- Missing usable metric value:
  - keep missing computed fields as null,
  - count as `skipped`,
  - do not emit conflict.
- Existing source lineage remains authoritative for preserved fields.

## Computation Rules (Deterministic)
- Sort order for rank computation:
  - primary: `metric value` descending,
  - secondary (deterministic tie break only): `journalId` ascending.
- Rank tie policy:
  - competition rank (`1, 1, 3`).
- Quartile (`quarter`) policy for missing quarter only:
  - split the sorted group into 4 position buckets (Q1..Q4) by count.
  - bucket boundaries use index ranges over group size `n`:
    - Q1: positions `1..ceil(n/4)`
    - Q2: positions `ceil(n/4)+1 .. ceil(n/2)`
    - Q3: positions `ceil(n/2)+1 .. ceil(3n/4)`
    - Q4: remaining positions
- `quartileRank` policy:
  - rank within the record's quartile subset using the same competition-rank tie policy.
- Inconsistency handling:
  - if preserved source `quarter` differs from bucket-derived quartile, treat as informational only (not error/conflict).

## Run Accounting Contract (For H18.2 Admin Execution)
- Required counters:
  - `processed`, `computed`, `preserved`, `skipped`, `errors`
- Required skip reason buckets:
  - `missing-metric-value`
  - `insufficient-grouping-data`
- Required behavior:
  - uncomputable due to missing metric value increments `skipped` only,
  - does not increment `errors`,
  - does not create conflict records.

Suggested internal summary DTO shape:

```json
{
  "processed": 0,
  "computed": 0,
  "preserved": 0,
  "skipped": 0,
  "errors": 0,
  "skipReasons": {
    "missing-metric-value": 0,
    "insufficient-grouping-data": 0
  }
}
```

## Acceptance Examples
- Tie handling:
  - values `[9.1, 9.1, 8.4]` -> ranks `[1, 1, 3]`.
- Quartile buckets (n=5):
  - positions `1-2` -> Q1, `3` -> Q2, `4` -> Q3, `5` -> Q4.
- Partial preservation:
  - source has `rank`, missing `quarter`/`quartileRank` -> keep `rank`, compute the two missing fields only.
  - source has `quarter`, missing `rank`/`quartileRank` -> keep `quarter`, compute missing fields only.
- Quarter mismatch:
  - source `quarter=Q2`, computed bucket says Q1 -> keep Q2; record informational mismatch only.
- Missing score:
  - no joined metric `value` -> leave missing fields null; count `skipped`; no conflict.

## H18.2 Unit-Test Checklist
- Deterministic rerun/idempotency on identical inputs.
- Stable tie ordering and rank outputs.
- Quartile bucket boundaries for uneven group sizes.
- Field-level preservation (no overwrite of source-provided fields).
- Source-quarter precedence in mismatch scenarios.
- Run summary counters and skip-reason buckets match expected outcomes.
