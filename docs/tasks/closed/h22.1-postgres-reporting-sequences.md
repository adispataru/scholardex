## 1. Ingest Write -> Projector Sync -> Steady-State Read

```mermaid
sequenceDiagram
    autonumber
    participant SRC as "Source Feed (Scopus/WoS)"
    participant MONGO as "Mongo Ingest Context"
    participant PROJ as "Postgres Projector"
    participant PG as "Postgres Read Context"
    participant APP as "Runtime Read Surfaces"

    SRC->>MONGO: Write ingest events/facts/lineage
    Note over MONGO: Mongo remains write-origin authority

    PROJ->>MONGO: Pull delta/full snapshot for projection
    MONGO-->>PROJ: Canonical ingest/read-source records
    PROJ->>PG: Upsert derived reporting read models

    APP->>PG: Read WoS/scoring/report/admin data
    PG-->>APP: SQL-backed read response
    Note over APP,PG: Post-cutover migrated reads are served from Postgres
```

## 2. Reset-First Backfill/Rebuild (Big-Bang Preparation)

```mermaid
sequenceDiagram
    autonumber
    participant OPS as "Operator"
    participant MONGO as "Mongo Ingest Context"
    participant PROJ as "Backfill/Rebuild Projector"
    participant PG as "Postgres Read Context"

    OPS->>PG: Reset/drop first-wave derived read datasets
    OPS->>PROJ: Start full rebuild run
    PROJ->>MONGO: Read full canonical ingest snapshot
    MONGO-->>PROJ: Snapshot data + lineage
    PROJ->>PG: Recreate/upsert full read model state
    PROJ-->>OPS: Rebuild summary (processed/failed/skipped)
```

## 4. Failure Branches and Recovery Triggers

```mermaid
flowchart TD
    A["Projector run starts"] --> B{"Mongo source reachable?"}
    B -- "No" --> B1["Fail run: projector lag/failure"]
    B1 --> B2["Keep pre-cutover runtime mode"]
    B2 --> B3["Alert + retry after source recovery"]

    B -- "Yes" --> C{"Projection/write to Postgres succeeds?"}
    C -- "No" --> C1["Fail run with partial progress metadata"]
    C1 --> C2["Rerun rebuild from reset/snapshot point"]
    C2 --> C3["Validate parity before any cutover"]

    C -- "Yes" --> D{"Parity/performance gates pass?"}
    D -- "No" --> D1["Abort cutover trigger"]
    D1 --> D2["Investigate drift/perf gap"]
    D2 --> D3["Fix projector/query model and rerun rebuild"]

    D -- "Yes" --> E["Execute big-bang cutover"]
    E --> F{"Post-cutover runtime errors exceed threshold?"}
    F -- "Yes" --> F1["Rollback trigger: revert read routing to pre-cutover state"]
    F1 --> F2["Rebuild/repair and retry cutover"]
    F -- "No" --> G["Steady-state Postgres read authority"]
```

## 5. Trigger Semantics (Locked for H22.1)

- Projector lag/failure:
  - does not change Mongo write-origin ownership.
  - blocks cutover until rebuild/parity checks pass.
- Rebuild rerun:
  - allowed and expected under reset-first strategy.
  - must converge to equivalent derived read state for the same source snapshot.
- Cutover abort/rollback:
  - cutover is canceled if parity/performance gates fail.
  - if post-cutover error budget breaches, runtime read routing must be reverted to pre-cutover mode while issues are fixed.
