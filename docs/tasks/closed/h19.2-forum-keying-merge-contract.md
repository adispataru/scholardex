# H19.2 Canonical Forum/Journal Keying and Merge Contract
Status: active indexed source-of-truth document.

## Summary
This document locks deterministic keying and merge behavior for cross-source forum/journal identity linking between WoS and Scopus in Scholardex.

Primary objective:
- resolve WoS journal identity and Scopus forum identity into `scholardex.forum_facts` deterministically;
- preserve additive source lineage via `scholardex.source_links`;
- quarantine ambiguity in `scholardex.identity_conflicts` instead of auto-merging.

Scope lock:
- contract/documentation only;
- implementation lands in H19.3+ and WoS onboarding subtasks.

## Inputs
- Source identity inputs:
  - `SCOPUS`: `source_id`, forum `issn`, `eIssn`, `publicationName`, `aggregationType`.
  - `WOS`: journal identity (`primaryIssn`, `eIssn`, `aliasIssns`, `title`/`normalizedTitle`).
  - `GSCHOLAR` and `USER_DEFINED`: optional forum identifiers and/or metadata for future linking under the same policy.
- Canonical storage targets:
  - `scholardex.forum_facts`
  - `scholardex.source_links`
  - `scholardex.identity_conflicts`

## Normalization
- ISSN canonical storage/match form:
  - uppercase dashed token `NNNN-NNNX` (for example `1234-567X`).
  - accepted raw variants include undashed forms, mixed case, and extra spaces.
  - invalid ISSN values are excluded from identity keys and flagged with conflict reason `NORMALIZATION_INVALID_ISSN` when needed.
- Forum name normalization:
  - lowercase,
  - trim and collapse whitespace,
  - punctuation-insensitive fingerprint for matching.
- Aggregation type normalization:
  - trimmed canonical token,
  - case-insensitive matching.

## Deterministic Keying
- Canonical forum IDs are deterministic fingerprints; UUID/random identifiers are forbidden.
- Key material precedence:
  1. sorted normalized ISSN token set (`issn`, `eIssn`, aliases).
  2. fallback `nameNormalized + aggregationTypeNormalized` when no valid ISSN exists.
- Source-to-canonical mapping key:
  - `(entityType=FORUM, source, sourceRecordId)` in `scholardex.source_links`.

## Deterministic Linking
- Step 1:
  - if `scholardex.source_links` already contains `(FORUM, source, sourceRecordId)`, reuse linked canonical forum id.
- Step 2:
  - candidate lookup by normalized ISSN overlap against canonical forum ISSN/alias set.
- Step 3:
  - if no valid ISSN key exists, candidate lookup by normalized `name + aggregationType`.
- Outcome rules:
  - `0` candidates: create canonical forum fact + `LINKED` source-link row.
  - `1` candidate: attach source link + additive merge into canonical forum fact.
  - `>1` equal-strength candidates: emit `OPEN` conflict, no canonical mutation.

Legacy ambiguity policy retirement:
- no recency-based auto-pick for ambiguous cross-source forum linking in H19.2 contract path;
- unresolved ambiguity must quarantine.

## Merge Rules
- Source identity envelopes are additive only:
  - `scopusForumIds[]`, `wosForumIds[]`, `googleScholarForumIds[]`, `userSourceForumIds[]`.
- Canonical display fields precedence:
  - Scopus preferred when available,
  - WoS fallback when Scopus value is absent,
  - other sources enrich aliases/lineage without destructive overwrite.
- Non-owned destructive overwrite is forbidden.
- Alias preservation is mandatory:
  - keep merged ISSN aliases and alternative names with lineage-safe updates.

## Conflict Rules
- Ambiguous forum identity resolution is quarantined in `scholardex.identity_conflicts`.
- Required forum conflict reason codes:
  - `AMBIGUOUS_ISSN_MATCH`
  - `AMBIGUOUS_NAME_AGG_MATCH`
  - `SOURCE_ID_COLLISION`
  - `NORMALIZATION_INVALID_ISSN`
- Conflict handling semantics:
  - status starts at `OPEN`,
  - canonical forum facts are non-mutating while unresolved,
  - source-link entry for the incoming source record may be marked `CONFLICT`.

## Compatibility
- Runtime lookup compatibility remains:
  - primary: canonical forum `id`,
  - compatibility: source forum ids through `scholardex.source_links`.
- Traversal compatibility/performance note:
  - forum linkage in H19 remains publication-driven;
  - no forum reverse-link arrays are required in this slice.
- Existing source-specific forum stores remain ingestion inputs during migration and are not final identity authority.
