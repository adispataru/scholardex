# H24.1 `/api/rankings/wos` Postgres Query Contract

Date: 2026-03-13  
Status: active contract baseline for the WoS rankings API SQL cutover.

## Summary

H24.1 locks the public query contract for `GET /api/rankings/wos` before runtime cutover from Mongo-backed `WosRankingView` reads to PostgreSQL `reporting_read.wos_ranking_view`.

This slice is documentation and decision-locking only. Runtime SQL implementation remains in H24.2 and H24.3.

## Decision Locks

- Public route remains `GET /api/rankings/wos`.
- Public response type remains `WosRankingPageResponse`.
- No auth or payload shape changes are introduced in H24.1.
- SQL cutover must preserve the current public request/response behavior unless a future task explicitly records a contract change.
- Tied-row ordering remains unspecified as part of the public contract to preserve current Mongo parity.

## Request Contract

Query parameters:
- `page` default `0`
- `size` default `25`
- `sort` default `name`
- `direction` default `asc`
- `q` optional

Validation envelope:
- `page >= 0`
- `1 <= size <= 100`
- `q.length <= 100` after request binding and before query execution
- invalid `sort` remains `400 bad_request`
- invalid `direction` remains `400 bad_request`

Allowed sort values only:
- `name`
- `issn`
- `eIssn`

Allowed direction values only:
- `asc`
- `desc`

## Query Normalization

`q` handling:
- `q == null` means no filter.
- `q.trim().isEmpty()` means no filter.
- otherwise `q.trim()` is the raw normalized query input for downstream search normalization.

Name normalization:
- trim leading/trailing whitespace,
- lowercase using root locale,
- collapse internal whitespace to a single space.

ISSN normalization:
- trim leading/trailing whitespace,
- uppercase using root locale,
- remove hyphens,
- remove spaces.

The same raw `q` contributes:
- a text-normalized probe for name search,
- an ISSN-normalized probe for ISSN, eISSN, and alternative ISSN search,
- both probes when both normalizations produce a non-blank value.

## Search Semantics

Search remains prefix-only, not contains-based.

Filter behavior:
- no `q` filter when normalized `q` is absent,
- otherwise rows match when any of the following prefixes match:
  - `name_norm`
  - `issn_norm`
  - `e_issn_norm`
  - `alternative_issns_norm`

Logical composition:
- OR across the canonical normalized search columns,
- text-normalized probe applies to `name_norm`,
- ISSN-normalized probe applies to `issn_norm`, `e_issn_norm`, and `alternative_issns_norm`.

Non-goal for H24.2:
- do not broaden search to `%contains%`,
- do not switch to tokenized/fuzzy search,
- do not add new searchable fields.

## Paging And Ordering Contract

Paging:
- requested page uses zero-based indexing,
- `totalItems` counts rows after applying the same filter,
- `totalPages = ceil(totalItems / size)`,
- empty result returns:
  - `items = []`
  - `totalItems = 0`
  - `totalPages = 0`

Ordering:
- public ordering is defined only by the requested primary sort field and direction,
- tied-row order is intentionally unspecified to preserve legacy Mongo behavior,
- H24.2 may use an internal deterministic tie-breaker for SQL paging stability, but that tie-breaker is non-contractual and must not change the allowed public sort fields.

## Response Compatibility

Response envelope remains:
- `items`
- `page`
- `size`
- `totalItems`
- `totalPages`

List item shape remains:
- `id`
- `name`
- `issn`
- `eIssn`
- `alternativeIssns`

Postgres mapping rule:
- `reporting_read.wos_ranking_view.journal_id` maps directly to API `id`.

No additional response fields are introduced in H24.1.

## H24.2 Implementation Guidance

The SQL adapter for H24.2 must:
- read only from `reporting_read.wos_ranking_view`,
- preserve the current route, auth contract, validation behavior, and response shape,
- use the canonical normalized columns:
  - `name_norm`
  - `issn_norm`
  - `e_issn_norm`
  - `alternative_issns_norm`
- preserve prefix-search behavior rather than switching to contains search,
- keep `name`, `issn`, and `eIssn` as the only public sort keys,
- preserve current error messages for invalid `sort`, `direction`, and `q` length where practical so controller/API tests remain stable.

## H24.4 Coverage Targets

H24.4 regression and parity coverage must include:
- default request contract,
- invalid `page`,
- invalid `size`,
- invalid `sort`,
- invalid `direction`,
- invalid overlength `q`,
- prefix-search parity for:
  - name,
  - ISSN,
  - eISSN,
  - alternative ISSN,
- response-shape parity,
- authenticated access,
- unauthenticated `401`,
- regression guard against accidental contains-search drift,
- regression guard against accidental reintroduction of Mongo-backed reads for this API surface.

## Assumptions

- `reporting_read.wos_ranking_view` is projection-populated before H24.2 runtime cutover is enabled.
- The table contains the normalized fields needed to preserve current search semantics.
- H24.1 does not change implementation code paths; it only locks the contract used by H24.2-H24.4.
