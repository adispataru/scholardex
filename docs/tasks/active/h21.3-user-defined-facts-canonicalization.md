# H21.3 USER_DEFINED Facts Canonicalization and Lineage Alignment

## Scope

This document locks H21.3 implementation behavior for source-specific `USER_DEFINED` fact stores and canonicalization into Scholardex facts used by existing projections.

H21.3 delivers:
- dedicated source fact stores:
  - `user_defined.publication_facts`
  - `user_defined.forum_facts`
- dedicated USER_DEFINED fact builder from `scopus.import_events` publication events,
- dedicated USER_DEFINED canonicalization into:
  - `scholardex.publication_facts`
  - `scholardex.forum_facts`
- deterministic source links for publication/forum lineage with `source=USER_DEFINED`,
- review/moderation metadata persistence and propagation without a separate approval workflow.

## Locked Decisions

- Wizard route and UI flow remain unchanged at `/user/publications/add`.
- H21.2 import-events path remains active; USER_DEFINED ingest still writes `scopus.import_events`.
- USER_DEFINED facts are source inputs only. Canonical Scholardex facts remain projection inputs.
- No direct `user_defined.*_facts -> scholardex.*_view` bypass is introduced.
- No historical backfill/replay migration is included in H21.3.
- Existing selected forum IDs are preserved; new wizard-created forums canonicalize via deterministic forum keying.

## Canonicalization Rules

- Forum canonicalization precedence aligns with H19.2:
  - match by ISSN/eISSN first (including aliases),
  - fallback to normalized forum name + normalized aggregation type.
- Ambiguous forum matches are quarantined (source-link conflict), not destructively merged.
- Publication canonicalization keeps deterministic ID material with `userSourceId` populated from `USER_DEFINED:PUBLICATION:*`.
- Author/affiliation linking behavior:
  - canonical IDs selected by wizard are treated as already resolved,
  - unresolved publication-author-affiliation links are quarantined via conflict-state source-link records.

## Review/Moderation Metadata

- Source facts persist explicit review metadata:
  - `approved`,
  - `reviewState`,
  - `reviewReason`,
  - `reviewStateUpdatedAt`,
  - `reviewStateUpdatedBy`,
  - `moderationFlow`,
  - submitter metadata (`wizardSubmitterEmail`, `wizardSubmitterResearcherId`, `wizardSubmittedAt`).
- Canonical publication/forum facts carry the same metadata for triage visibility.
- Workflow semantics remain metadata-only in H21 (`no separate admin approval UI/workflow`).
