# H21.1 USER_DEFINED Wizard-Onboarding Contract

Date: 2026-03-14  
Status: active decision-locked contract for H21.1.

## 1. Summary

This document locks the H21.1 contract for migrating the existing publication wizard onboarding path into canonical `USER_DEFINED` ingestion.

Locked direction:

- keep the existing wizard route family at `/user/publications/add` (in-place migration only),
- lock wizard ingest source family to `USER_DEFINED`,
- lock deterministic wizard publication/forum source record key shapes under `USER_DEFINED` prefixes,
- align forum keying precedence with H19.2,
- require explicit review/moderation metadata semantics without introducing a separate admin approval workflow.

This is a contract-and-enforcement slice. Runtime submit-path migration remains in H21.2+.

## 2. Source And Keying Contract

- Canonical source token for wizard onboarding: `USER_DEFINED`.
- Legacy source alias `USER_PUBLICATION_WIZARD` is compatibility input only and must normalize to `USER_DEFINED` in source-link lineage handling.
- Deterministic forum source-record shape:
  - prefix: `USER_DEFINED:FORUM:`
  - key precedence:
    1. normalized `issn/eIssn` pair (H19.2 policy),
    2. fallback normalized `publicationName + aggregationType` when no valid ISSN exists.
- Deterministic publication source-record shape:
  - prefix: `USER_DEFINED:PUBLICATION:`
  - key precedence:
    1. normalized DOI when present,
    2. fallback normalized tuple: `title + coverDate + creator + forumSourceRecordId`.

Hashing/determinism requirements:

- key generation must be stable for equivalent normalized inputs,
- whitespace/case/noise normalization must not change deterministic output,
- no random/UUID-based wizard source-record IDs.

## 3. Lineage And Metadata Contract

Required lineage envelope fields for wizard-ingested canonical records:

- `source`
- `sourceRecordId`
- `sourceEventId`
- `sourceBatchId`
- `sourceCorrelationId`

Review/moderation metadata semantics for H21:

- wizard submissions are explicitly marked for operator review metadata (`PENDING_OPERATOR_REVIEW`),
- moderation is metadata/state-based only in H21,
- no separate admin approval UI/workflow is introduced by H21.

## 4. Scope Locks And Non-Goals

- H21.1 must not introduce new wizard route families beyond `/user/publications/add`.
- H21.1 must not expand scope into standalone direct author/forum/affiliation onboarding projects.
- H21.1 must not migrate runtime wizard submit behavior; that implementation lands in H21.2+.
