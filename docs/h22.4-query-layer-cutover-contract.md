# H22.4 Query-Layer Cutover Contract (Mongo -> Postgres Reads)

## Summary
H22.4 introduces a single runtime read-store selector for first-wave reporting/admin read surfaces:
- `app.reporting.read-store=mongo|postgres`
- default remains `mongo`
- `postgres` mode activates SQL-backed read adapters for WoS/reporting and Scopus author/forum/affiliation/admin read surfaces

Mongo remains authoritative for ingest/event/canonical writes.

## Cutover Scope
The following surfaces switch together under the same selector:
- `ReportingLookupPort` (via `SwitchableReportingLookupFacade`)
- `ScholardexAuthorQueryService`
- `ScholardexForumQueryService`
- `ScholardexAffiliationQueryService`
- `ScholardexAdminReadFacade`

Each surface routes to Mongo adapters in `mongo` mode and Postgres adapters in `postgres` mode.

## Startup Guard (All-Or-Nothing)
`PostgresReadCutoverGuard` enforces readiness when selector is `postgres`:
- requires H22.4 Postgres adapters to be present in the application context
- requires H22.3 projection checkpoints for both slices:
  - `wos`
  - `scopus`

If readiness checks fail, startup aborts with explicit error.

## SQL Read Contract
Postgres adapters query only `reporting_read.*` objects from H22.2/H22.3.
No endpoint payload contract changes are introduced in H22.4.

## Rollback Contract
Rollback is configuration-only:
- set `app.reporting.read-store=mongo`
- restart application

No data rollback is required because H22.4 changes only read path selection.
