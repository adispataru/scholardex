# H19.1 Multi-Source Scholardex Identity and Ownership Contract

## Summary
This document locks the canonical multi-source identity and ownership contract for H19.1.

Operational policy lock:
- Scholardex is the authoritative identity layer for publication, author, forum, and affiliation entities.
- Supported source families are `SCOPUS`, `WOS`, `GSCHOLAR`, and `USER_DEFINED`.
- Source-specific events/facts remain ingestion-authoritative inputs and lineage roots.
- Runtime merged projections are derived from authoritative Scholardex facts.

Contract scope lock:
- This is a contract/documentation slice only.
- H19.2+ implement schema, ingestion, linking, and runtime cutover behavior based on this contract.

## Scope
This contract covers:
- canonical Scholardex identity entities:
  - publications
  - authors
  - forums
  - affiliations
  - citations
- identity keying and idempotence rules,
- source ownership boundaries,
- merge and conflict quarantine semantics,
- compatibility lookup behavior,
- migration guardrails for post-H19 cutover.

Out of scope for H19.1 implementation:
- concrete migration scripts,
- runtime endpoint rewires,
- full source adapters for WoS/Scholar/User ingestion.

## Canonical Collections and Views

### `scholardex.publication_facts`
- Purpose: authoritative canonical publication identity and cross-source link state.
- Required fields:
  - `id` (`scholardexPublicationId`, deterministic canonical primary id)
  - source identity envelope:
    - `eid` (Scopus key, optional)
    - `wosId` (WoS key, optional)
    - `googleScholarId` (Scholar key, optional)
    - `userSourceId` (user-defined source key, optional)
  - compatibility fields:
    - `doi`
    - `doiNormalized`
    - `title`
    - `titleNormalized`
    - `forumId`
    - `authorIds`
    - `affiliationIds`
    - `coverDate`
  - lineage fields:
    - `sourceEventId`
    - `source`
    - `sourceRecordId`
    - `sourceBatchId`
    - `sourceCorrelationId`
    - `createdAt`
    - `updatedAt`
- Canonical identity key:
  - deterministic fingerprint over normalized publication identity material
    (source IDs when present, otherwise normalized DOI/title/date/forum linkage keys).
- Idempotence/upsert uniqueness key:
  - canonical `id` plus source-key uniqueness constraints on non-null source IDs.

### `scholardex.author_facts`
- Purpose: authoritative canonical author identity independent of source-specific ids.
- Required fields:
  - `id` (`scholardexAuthorId`, deterministic canonical primary id)
  - source identity envelope:
    - `scopusAuthorIds[]`
    - `wosAuthorIds[]`
    - `googleScholarAuthorIds[]`
    - `userSourceAuthorIds[]`
  - identity fields:
    - `displayName`
    - `nameNormalized`
    - `affiliationIds`
  - lineage fields:
    - `sourceEventId`
    - `source`
    - `sourceRecordId`
    - `sourceBatchId`
    - `sourceCorrelationId`
    - `createdAt`
    - `updatedAt`
- Canonical identity key:
  - deterministic fingerprint over normalized source-author key set and normalized identity hints.
- Idempotence/upsert uniqueness key:
  - canonical `id` plus unique non-null source author ids.

    - `userSourceForumIds[]`
  - identity fields:
    - `name`
    - `nameNormalized`
    - `issn`
    - `eIssn`
    - `aliasIssns[]`

  - lineage fields:
    - `sourceEventId`
    - `source`
    - `sourceRecordId`
    - `sourceBatchId`
    - `sourceCorrelationId`
    - `createdAt`
    - `updatedAt`
- Canonical identity key:
  - deterministic fingerprint over normalized ISSN/eISSN aliases and normalized forum identity tokens.
- Idempotence/upsert uniqueness key:
  - canonical `id` plus unique non-null source forum ids.

### `scholardex.publication_view`
- Purpose: runtime merged projection derived from authoritative Scholardex facts plus source-owned enrichments.
- Contract position:
  - derived view only; not authoritative identity storage.
  - must preserve compatibility keys used by runtime reads (`id`, `eid`, `wosId`, `googleScholarId`).

### `scholardex.citation_facts`
- Purpose: authoritative canonical cross-source citation edges.
- Why:
  - source-specific EID-only citation edges are insufficient for WoS-only, Scholar-only, or user-defined publications.
- Required fields:
  - `id`
  - `citedPublicationId` (canonical scholardex publication id)
  - `citingPublicationId` (canonical scholardex publication id)
  - `source`
  - `sourceRecordId`
  - `sourceEventId`
  - `sourceBatchId`
  - `sourceCorrelationId`
  - `createdAt`
  - `updatedAt`
- Canonical identity key:
  - `(citedPublicationId, citingPublicationId, source)`
- Idempotence/upsert uniqueness key:
  - `(citedPublicationId, citingPublicationId, source)`

### `scholardex.identity_conflicts`
- Purpose: generic cross-entity conflict quarantine for canonical identity merge/link operations.
- Required fields:
  - `id`
  - `entityType` (`PUBLICATION|FORUM|AUTHOR|AFFILIATION`)
  - `incomingSource`
  - `incomingSourceRecordId`
  - `candidateCanonicalIds[]`
  - `reasonCode`
  - `status` (`OPEN|RESOLVED|DISMISSED`)
  - lineage fields:
    - `sourceEventId`
    - `sourceBatchId`
    - `sourceCorrelationId`
  - `detectedAt`
  - `resolvedAt`
  - `resolvedBy`
- Canonical identity key:
  - conflict record id.
- Idempotence/upsert uniqueness key:
  - `(entityType, incomingSource, incomingSourceRecordId, reasonCode, status=OPEN)` for open-conflict dedupe.

### `scholardex.source_links`
- Purpose: deterministic source-record to canonical-entity mapping ledger for replay, traceability, and debugging.
- Required fields:
  - `id`
  - `entityType` (`PUBLICATION|FORUM|AUTHOR|AFFILIATION|CITATION`)
  - `source`
  - `sourceRecordId`
  - `canonicalEntityId`
  - `linkState` (`LINKED|CONFLICT|UNMATCHED|SKIPPED`)
  - `linkReason`
  - lineage fields:
    - `sourceEventId`
    - `sourceBatchId`
    - `sourceCorrelationId`
  - `linkedAt`
  - `updatedAt`
- Canonical identity key:
  - `(entityType, source, sourceRecordId)`
- Idempotence/upsert uniqueness key:
  - `(entityType, source, sourceRecordId)`

## Field Ownership Matrix (Locked)
- Scopus-owned source fields:
  - Scopus source ids and Scopus bibliographic payload fields.
- WoS-owned source fields:
  - WoS source ids and WoS-specific ranking/classification enrichments.
- Scholar-owned source fields:
  - Scholar source ids and Scholar-specific profile/citation enrichments.
- User-defined source fields:
  - user-defined source ids and user-provided metadata under moderation/approval policy.
- Scholardex-owned canonical fields:
  - canonical deterministic ids (`scholardex*Id`),
  - normalized identity keys,
  - source-id envelopes,
  - cross-source linkage state,
  - canonical relationship edges among publication/author/forum/affiliation.

Ownership rules:
- Source-specific ingestors may write only source-owned input fields + lineage.
- Canonical linker/merger may write Scholardex-owned canonical fields.
- Researcher linkage fields are profile/account ownership and must not be stored in `scholardex.author_facts`.
- Cross-source destructive overwrite of non-owned fields is forbidden.
- Enrichment is additive and ownership-safe.

## Identity, Replay, and Merge Rules
- Event identity fields are mandatory for all canonical writes:
  - `source`, `sourceRecordId`, `batchId`, `correlationId`, `payloadHash`.
- Canonical IDs are deterministic and fingerprint-based; random UUID canonical IDs are forbidden.
- Replay of identical source event sets must produce identical canonical identity state.
- Best-effort merge is required:
  - perform deterministic merge attempts using normalized identity keys,
  - additive enrichment when source identity resolves to an existing canonical entity,
  - no blind overwrite of existing non-owned values.
- WoS/Scopus nuance lock:
  - WoS contributes minimal identity + ranking context.
  - Scopus may enrich existing canonical entities with additional metadata (for example forum type details) when deterministic match exists.
  - unresolved deterministic match must emit conflict, not force merge.

## Conflict Rules (Locked)
- Ambiguous or colliding identity resolution must be quarantined.
- Conflict records must include:
  - entity type,
  - incoming source/sourceRecordId,
  - conflicting candidate canonical ids,
  - rule/reason code,
  - resolver status (`OPEN|RESOLVED|DISMISSED`),
  - lineage fields (`sourceEventId`, `sourceBatchId`, `sourceCorrelationId`),
  - timestamps.
- Conflict handling is non-mutating for authoritative canonical entities until resolution policy allows deterministic apply.
- Legacy conflict stores may remain for backward compatibility during migration, but H19 target is unified handling via `scholardex.identity_conflicts`.

## Compatibility and Migration Guardrails
- Lookup compatibility contract:
  - primary lookup key is canonical `id`,
  - compatibility keys include source ids (`eid`, `wosId`, `googleScholarId`, `userSourceId`).
- Runtime must normalize compatibility keys to canonical ids before downstream processing.
- Post-H19 cutover guardrail:
  - runtime write paths must not write identity state directly to source-silo identity stores.
  - source-specific collections are ingestion inputs and lineage history, not final runtime identity authority.
- Citation compatibility note:
  - canonical citation runtime must resolve by canonical publication ids;
  - source/EID-keyed citation stores are transitional inputs only after cutover.
- Researcher compatibility note:
  - existing `Researcher` source-id fields (`scopusId[]`, `wosId[]`, `scholarId`) remain valid in H19.1.
  - these source ids are not moved into `scholardex.author_facts` as researcher linkage fields in this slice.

## Researcher and ScholardexAuthor Contract
- `Researcher` remains a separate profile/account-linked entity.
- `ScholardexAuthor` is the canonical multi-source scholarly identity.
- `Researcher` stores a single canonical link field: `primaryScholardexAuthorId`.
- Link creation points in this contract slice are onboarding and group creation workflows.
- `scholardex.author_facts` must not carry researcher ids.
- No implicit one-to-one assumption beyond this explicit canonical link field is allowed.
