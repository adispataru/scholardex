# H22.3 Postgres Projection/Sync Contract

Date: 2026-03-11  
Status: active implementation baseline for Mongo-canonical -> Postgres reporting projection.

## Summary

H22.3 delivers an admin-triggered projector that materializes `reporting_read` PostgreSQL tables from canonical Mongo collections with deterministic, replay-safe behavior.

This slice does not change runtime read facades. SQL read cutover remains in H22.4.

## Slice Boundaries

Two projection slices are enforced:

- `wos`
  - targets: `wos_ranking_view`, `wos_metric_fact`, `wos_category_fact`, `wos_scoring_view`
  - source collections: `wos.ranking_view`, `wos.metric_facts`, `wos.category_facts`, `wos.scoring_view`
- `scopus`
  - targets: `scholardex_publication_view`, `scholardex_author_view`, `scholardex_forum_view`, `scholardex_affiliation_view`, `scholardex_citation_fact`, `scholardex_authorship_fact`, `scholardex_author_affiliation_fact`
  - source collections: `scholardex.publication_view`, `scholardex.author_view`, `scholardex.forum_view`, `scholardex.affiliation_view`, `scholardex.citation_facts`, `scholardex.authorship_facts`, `scholardex.author_affiliation_facts`

## Fingerprint Rules

Incremental sync uses per-slice source fingerprints:

- `wos`: stable hash of counts plus max `updatedAt`/`createdAt` and max `sourceVersion` markers across WoS source collections.
- `scopus`: stable hash of counts plus max `updatedAt`/`buildAt` (or `createdAt` for edge collections) across Scopus source collections.

If fingerprint is unchanged versus checkpoint, the slice is skipped.

## Checkpoint and Run State

Projection state tables in `reporting_read`:

- `projection_checkpoint`: last successful fingerprint per slice (`wos`, `scopus`) and last run metadata.
- `projection_run`: run-level status (`RUNNING`, `SUCCESS`, `FAILED`) and timing/error summary.
- `projection_slice_run`: per-slice outcome (`SUCCESS`, `SKIPPED`, `FAILED`) and inserted-row counts.

Reset operation (`/admin/initialization/postgres/projection/resetState`) clears checkpoints only.

## Write and Failure Contract

- Write mode is **slice replace via staging temp tables** in a single SQL transaction per slice.
- Ordering is FK-safe:
  - WoS facts/views: truncate dependents first, insert parent rows first.
  - Scopus facts/views: truncate edge facts first, insert entity views first.
- Projector is **fail-fast**:
  - first hard failure marks run/slice failed,
  - failing slice checkpoint is not advanced,
  - previously successful slice checkpoints from the same run remain committed.

## Determinism and Lineage

- Source reads are sorted by canonical IDs/keys before staging writes.
- Projected view rows use projector run metadata (`build_version`, `build_at`, `updated_at`) for reproducible lineage.
- Existing source lineage columns (for example `source_event_id`, `scopus_lineage`, `wos_lineage`, `scholar_lineage`) are preserved from Mongo source documents.

## Admin Surface

Added initialization endpoints:

- `POST /admin/initialization/postgres/projection/runFull`
- `POST /admin/initialization/postgres/projection/runIncremental`
- `POST /admin/initialization/postgres/projection/showStatus`
- `GET /admin/initialization/postgres/projection/status`
- `POST /admin/initialization/postgres/projection/resetState`

## Out of Scope

- SQL read-path cutover (`H22.4`).
- Materialized view refresh policy (`H22.5`).
- parity/performance gating automation (`H22.6`).
- production runbook hardening (`H22.7`).
